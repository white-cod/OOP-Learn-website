Інкапсуляція

Одним із найважливіших факторів при проектуванні компонентів програми є приховування внутрішніх даних компонента та деталей його реалізації від інших компонентів програми та надання набору методів для взаємодії з ним (API). Цей принцип є одним із чотирьох фундаментальних принципів ОOП і називається інкапсуляцією.

Правильна інкапсуляція важлива з багатьох причин:

1. Вона сприяє перевикористання компонентів: оскільки в цьому випадку компоненти взаємодіють один з одним лише за допомогою їх API і байдужі до змін внутрішньої структури, вони можуть використовуватися у ширшому контексті

2. Інкапсуляція прискорює процес розробки: слабко пов'язані один з одним компоненти (тобто компоненти, код яких якнайменше звертається або використовує код інших компонентів) можуть розроблятися, тестуватися і доповнюватися незалежно.

3. Правильно інкапсульовані компоненти легші для розуміння та процесу налагодження, що спрощує підтримку програми. У мові Java інкапсуляція реалізована за допомогою системи класів, що дозволяють зібрати інформацію про об'єкт в одному місці; пакетів, які групують класи за будь-яким критерієм, та модифікаторів доступу, якими можна помітити весь клас або його поле чи метод.

Усього модифікаторів доступу чотири:


● public – повний доступ до сутності (поля чи методу класу) з будь-якого пакету;

● protected – доступ до сутності лише для класів свого пакета та спадкоємців класу;

● неявний модифікатор за промовчанням (за відсутності трьох явних) – доступ до сутності лише класів свого пакета;

● private – доступ лише усередині класу, у якому оголошено сутність.

Для досягнення правильної інкапсуляції необхідно надати коректний API для роботи з компонентом. Наприклад, в сеттер для змінної можна включити логіку з перевірки значень, що передаються, або не надавати сеттери в класі зовсім, якщо клас повинен бути доступний тільки для читання.

C++
class BankAccount {
	private:
		std::string accountNumber;
		double balance;
	public:
		void setAccountNumber(std::string number) {
				accountNumber = number;
			}
			std::string getAccountNumber() {
				return accountNumber;
			}
			void setBalance(double amount) {
				balance = amount;
			}
			double getBalance() {
				return balance;
			}
			void deposit(double amount) {
				balance += amount;
			}
			void withdraw(double amount) {
				if (amount > balance) {
					std::cout << "Insufficient funds!" << std::endl;
				} else {
					balance -= amount;
				}
			}
		};
					
int main() {
	BankAccount account;
	account.setAccountNumber("123456");
	account.setBalance(1000.0);
	std::cout << "Account number: " << account.getAccountNumber() << std::endl;
	std::cout << "Balance: " << account.getBalance() << std::endl;
	account.deposit(500.0);
	std::cout << "New balance after deposit: " << account.getBalance() << std::endl;
	account.withdraw(2000.0);
	std::cout << "New balance after withdrawal: " << account.getBalance() << std::endl;
	return 0;
}
Terminal
Account number: 123456
Balance: 1000
New balance after deposit: 1500
Insufficient funds!
New balance after withdrawal: 1500

Принцип роботи

Принцип інкапсуляції в цій програмі полягає в тому, що дані, пов’язані з банківським рахунком (номер рахунку та баланс), зберігаються в приватних змінних у класі BankAccount. Ці змінні не доступні безпосередньо ззовні класу. Замість цього надаються загальнодоступні методи (такі як setAccountNumber(), getAccountNumber(), setBalance(), getBalance(), deposit() і reject()) для доступу та зміни цих приватних змінних.

Це гарантує захист даних від випадкової або зловмисної модифікації поза класом. Методи також забезпечують рівень абстракції, що дозволяє змінювати реалізацію класу, не впливаючи на код, який його використовує. Наприклад, реалізацію методів депозиту() і знімання() можна змінити, щоб додати додаткову логіку або взаємодіяти із зовнішніми системами без зміни коду, який викликає ці методи.

Таким чином, інкапсуляція є фундаментальним принципом об’єктно-орієнтованого програмування, який сприяє приховуванню інформації та абстракції шляхом обмеження доступу до внутрішнього стану об’єктів і надання керованого інтерфейсу для взаємодії з ними.
